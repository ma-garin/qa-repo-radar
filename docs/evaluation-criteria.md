# Evaluation Criteria

## Core policy

Repositories must be evaluated from the viewpoint of third-party software testing and quality assurance service companies.

A repository is valuable when it improves one or more of the following:

- customer-facing QA service delivery
- test design quality
- test automation efficiency
- quality risk visibility
- verification PMO capability
- QA process improvement
- standardization and reuse
- AI quality assurance capability
- internal training and technical enablement

Popularity alone is not sufficient.

## Scoring dimensions

| Dimension | Max points | Description |
|---|---:|---|
| Third-party QA applicability | 30 | Can be used in services similar to third-party verification, QA consulting, or testing support |
| Customer explanation value | 25 | Helps explain quality, risk, coverage, test strategy, or improvement value to customers |
| Test design usefulness | 25 | Supports test design, coverage modelling, pairwise, decision tables, MBT, or test generation |
| Quality risk reduction | 20 | Helps reduce escaped defects, regressions, flaky tests, quality blind spots, or release risk |
| Automation integration | 20 | Can be integrated into CI/CD, automated testing, or repeatable QA workflows |
| Standardization potential | 20 | Can become a reusable internal method, template, framework, or practice |
| Documentation quality | 15 | README, examples, docs, or tutorials are sufficient for evaluation |
| License clarity | 15 | License is present and suitable for evaluation or business use consideration |
| Maintenance freshness | 10 | Updated recently enough to be worth evaluating |
| Adoption feasibility | 10 | Setup and usage are realistic for project teams |

## Penalties

| Condition | Points |
|---|---:|
| No clear QA or testing relevance | -30 |
| More than one year without meaningful updates | -30 |
| README is too thin to evaluate | -20 |
| Personal experiment with low reuse potential | -20 |
| Popular but not useful for QA service delivery | -20 |
| License missing or unclear | -15 |
| Requires heavy infrastructure before value can be confirmed | -10 |

## Classification labels

Use the following labels first:

- customer-proposal-ready
- internal-standardization-candidate
- automation-foundation
- test-design-support
- quality-visualization
- qa-pmo-support
- agile-qa-support
- ai-quality-assurance
- learning-reference
- watchlist
- low-priority

## Evidence rule

Every score should be explainable using observable evidence such as:

- repository description
- topics
- README content
- examples
- docs
- CI files
- license
- update date
- issue activity

Inferred judgement must be marked as inference in reports.
